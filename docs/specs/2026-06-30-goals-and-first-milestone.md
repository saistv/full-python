# Goals and First Milestone

## Objective

Build a Python-first research system that can find and validate a robust, operationally survivable NQ/MNQ intraday momentum-breakout edge. The system should use high-quality NQ market data for signal research, validate risk and sizing assumptions as MNQ-first, and preserve a complete audit trail for every bar, signal, rejection, order intent, stop update, exit, risk veto, and state transition.

The first objective is not live execution. The first objective is one reproducible, auditable baseline strategy report that can show whether the original strategy thesis is worth pursuing.

## Primary Goal

Find a statistically credible positive-expectancy strategy before trying to scale toward an income target.

The project should optimize for durable edge first, then survivability, then scalability. A high monthly profit target is treated as a future sizing question, not proof that the edge exists.

## Success Definition

A candidate is worth pursuing only if it is operationally survivable and not merely profitable in-sample.

The first-stage success standard is:

- maximum drawdown near or below $5,000 under MNQ-first sizing assumptions
- acceptable loss streaks for hands-off operation
- tolerable time underwater
- positive expectancy after commissions and realistic slippage
- no dependence on one month, one regime, or a few outlier trades
- acceptable performance after removing the top trades or top days
- stable behavior across nearby parameter values
- reasonable behavior across multiple market regimes and validation windows
- every accepted and rejected signal explainable from the event ledger

## Research Universe

The system should analyze all available NQ/MNQ data to understand behavior across RTH and ETH, but only RTH candidates are eligible for the first shadow, paper, or live promotion path.

RTH is the first promotion target because it is easier to monitor, reason about, and operationally survive. ETH and overnight behavior remain research inputs, not initial deployment targets.

## Instrument And Risk Assumption

Research should use NQ-quality data and signal behavior, but risk validation should be MNQ-first.

This separates market-quality research from account-survivability constraints:

- NQ-quality data is used for liquidity, signal behavior, and continuity.
- MNQ-first sizing is used for drawdown, loss streak, psychological survivability, and eventual live validation.

## Strategy Family

The first validated strategy family remains the original thesis, with defensive controls:

- intraday momentum breakout
- ATF trend-state core
- support/resistance breakout logic
- squeeze, prove-it, and candle-quality filters as candidate primitives
- rejected-trade analysis
- MFE/MAE path analysis
- defensive regime permissions
- dynamic entry-window research

The first milestone does not include a full hybrid system. Mean reversion, separate chop-regime strategies, order-flow features, and live adaptive switching are research-later topics.

## Chosen Approach

Use the balanced foundation approach.

This means the first milestone should include only enough infrastructure to produce one clean baseline report:

- canonical data loading
- deterministic replay
- append-only event ledger
- one baseline strategy implementation
- one survivability report

This avoids two failure modes:

- rushing into broad sweeps before the system can explain its decisions
- building a large architecture without early feedback on whether the strategy thesis is viable

## First Milestone Deliverable

The first milestone is one reproducible baseline strategy report generated by the Full Python project.

The report should include:

- data manifest
- strategy config
- run ID and parameter hash
- trade list
- rejected-signal list
- event ledger path
- equity curve
- drawdown
- max loss streak
- time underwater
- monthly breakdown
- long/short breakdown
- exit-reason breakdown
- MFE/MAE summary
- top-trade dependency
- costs and slippage assumptions

The report should be reproducible from source code, config, and data manifest.

## Architecture Boundary

The project should keep these boundaries clear:

- `data`: converts vendor files into canonical market bars
- `indicators`: computes reusable market features
- `strategy`: produces signal decisions and order intents
- `replay`: feeds bars through strategy deterministically
- `ledger`: records every event in append-only form
- `risk`: vetoes unsafe order intents
- `research`: runs experiments and survivability analysis
- `reporting`: produces human-readable run reports
- `execution`: remains adapter-only until shadow/paper phases

The strategy layer must not call broker or execution code directly. It returns decisions. Execution adapters act on decisions later.

## Non-Goals

Do not build these in the first milestone:

- direct Tradovate execution
- TradersPost adapter
- live order submission
- live add-on logic
- anti-martingale sizing
- auto-optimization that changes live parameters
- full mean-reversion strategy
- multi-account execution
- multi-broker routing
- wholesale port of old Pine code

The old Pine system is reference material, not the canonical source of truth.

## Promotion Gates

Research candidates may progress only through gates:

1. Baseline replay gate: deterministic replay produces the same result from the same data and config.
2. Research quality gate: candidate survives costs, slippage, nearby parameters, top-trade removal, and validation windows.
3. Shadow gate: Python generates live decisions without sending orders and logs all accepted/rejected signals.
4. Paper gate: Python submits paper orders with full reconciliation.
5. Limited live gate: smallest practical size, one instrument, strict risk controls, and manual review.

No candidate should move to live execution because of a single strong backtest.

## Next Implementation Plan

The next implementation plan should cover:

1. Canonical data loader boundary for historical NQ/MNQ files.
2. Data manifest model.
3. Baseline momentum-breakout strategy skeleton.
4. Rejected-signal event logging.
5. Baseline survivability report.

The plan should avoid broad optimization until the baseline report is reproducible and auditable.
